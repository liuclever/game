## 📦 模块依赖关系

```
interfaces/          (依赖 ↓)
    ↓
application/         (依赖 ↓)
    ↓
domain/              (不依赖任何层)
    ↑
infrastructure/      (实现 ↑)
    ↓
configs/             (被读取)
```

**依赖规则：**
- 上层可以依赖下层
- 下层不能依赖上层
- domain 层不依赖任何层（纯业务逻辑）
- infrastructure 层实现 domain 层定义的接口

## 🎯 核心模块说明

## 🔑 设计原则

### 1. 单一职责原则
每个模块只负责一种变化方向：
- domain：游戏规则变化
- application：业务流程变化
- interfaces：界面展示变化
- infrastructure：技术实现变化

### 2. 依赖倒置原则
- domain 定义接口，infrastructure 实现接口
- 高层模块不依赖低层模块，都依赖抽象

### 3. 开闭原则
- 对扩展开放：可以添加新功能
- 对修改关闭：不修改现有代码

## 💡 开发建议

### 添加新功能的步骤：

1. **定义实体** (domain/entities/)
   - 创建新的实体类

2. **定义规则** (domain/rules/)
   - 实现核心业务逻辑

3. **定义仓储接口** (domain/repositories/)
   - 定义数据访问接口

4. **实现仓储** (infrastructure/db/)
   - 实现数据库访问

5. **创建服务** (application/services/)
   - 串联规则，形成业务流程

6. **添加路由** (interfaces/web_api/routes/)
   - 暴露 API 接口

7. **开发前端** (interfaces/client/)
   - 实现用户界面

8. **配置数据** (configs/)
   - 添加游戏配置

9. **编写测试** (tests/)
   - 保证代码质量
